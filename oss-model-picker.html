Which open-weight model · and when

Default to a small local model. Scale up when the task demands it.

Ollama's library exposes dozens of open-weight models from Meta, Alibaba (Qwen), DeepSeek, Mistral, Google, Microsoft, OpenAI, Moonshot and Zhipu. You don't need to memorise them — you need a clear rule: start with a lightweight model that fits your hardware, then escalate or downshift based on the task. This page maps the familiar five-task taxonomy to recommended open-weight models, with hardware footprint and license overlaid on every card instead of per-token pricing — because locally, your currency is VRAM, not credits.

2 · Anti-patterns to call out

The five most expensive local-model mistakes

Expensive in hours and hardware rather than AI Credits — and how to avoid them.

⚠️ Frontier MoE pulled onto a laptop

ollama pull deepseek-v3.1:671b is a ~400 GB download that will swap-thrash any consumer machine into oblivion. If you want R1-class reasoning locally, use the distilled variants (deepseek-r1:8b / :14b / :32b) — or run the full model hosted and keep local for everything else.

⚠️ Default context window in agent loops

Ollama's default context is small (4K on many tags). Agent workflows silently truncate tool output and file contents, producing confidently wrong answers. Set OLLAMA_CONTEXT_LENGTH or num_ctx explicitly (32K+ for agents) — and budget the extra KV-cache memory that comes with it.

⚠️ Reasoning model on autocomplete-style tasks

Thinking models (DeepSeek-R1, Qwen3 in thinking mode) emit hundreds of reasoning tokens before answering — terrible latency for "rename this variable". Use a small coder model with fill-in-the-middle support (qwen2.5-coder:7b) for completions; save thinking models for genuinely hard debugging.

⚠️ Over-quantised weights, under-diagnosed quality

A 70B at Q2 usually loses to a 32B at Q4 while costing more memory. If quality drops, suspect the quantisation before the model family. Q4_K_M is the sane default; go Q8/FP16 only when you have headroom and a measured quality gap.

✅ Start small, then promote

Start every workflow on a Lightweight-tier model that fits comfortably in memory. Only promote to a bigger model when you can name the failure (missed multi-file dependency, weak reasoning chain, hallucinated API). This is the local equivalent of "use Auto, then promote" — it keeps your iteration loop fast and your GPU thermals sane.
